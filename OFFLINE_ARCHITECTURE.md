# داکیومنت معماری آفلاین و استراتژی پایداری داده‌های آکادمی توآمریکا (ToAmerica)

این سند تشریح‌کننده معماری آفلاین، مدیریت دیتابیس‌های محلی، همگام‌سازی امن (Offline State Management) و مدیریت دارایی‌های رسانه‌ای در وب‌سایت کشینگ هوشمند کمپانی **توآمریکا** همگام با پایداری حداکثری و امنیت بر‌تر مطابق با استانداردهای نوین فناوری وب است.

---

## ۱. اهداف اصلی معماری آفلاین (Offline Architecture Goals)
1. **دسترس‌پذیری ۱۰۰٪ (Always Available):** امکان بازبینی ویدیوهای درسی، فلوها، جزوات مهاجرتی و ثبت تیکت‌ها تحت شرایط نوسان یا قطع کامل اتصال شبکه.
2. **پایداری داده‌ها (Data Integrity):** ممانعت از نابودی اطلاعات ورودی کاربر در فرم‌های مشاوره و ثبت‌نام با بهره‌گیری از حائل ذخیره‌سازی محلی (LocalStorage / IndexedDB).
3. **امنیت پیشرفته (Local Data Hardening):** رمزنگاری تمامی کلیدهای هویتی و کوئری‌های تعاملاتی کاربر با الگوریتم‌های استاندارد متقارن پیش از ذخیره‌سازی متمم.
4. **شبیه‌سازی یکپارچه (Zero External Requests):** عدم اتکای برنامه به بسته‌های شبکه ثالث (CDN) و قرارگیری بومی تمام رفرنس‌ها در داخل دایرکتوری نهایی ساخت (Vite Bundle).

---

## ۲. طراحی لایه‌های ذخیره‌سازی محلی (Local Storage Tiering)

جهت تضمین پایداری تحت هر شرایط، ساختار ذخیره‌سازی به سه لایه مجزا دسته‌بندی می‌گردد:

```
┌────────────────────────────────────────────────────────────────────────┐
│                        مرورگر کاربر (Browser Client)                   │
├────────────────────────────────────────────────────────────────────────┤
│  ┌────────────────────────┐  ┌──────────────────┐  ┌────────────────┐  │
│  │   محفظه کش استاتیک     │  │    دیتابیس محلی  │  │  تنظیمات محلی  │  │
│  │  (Cache Storage API)   │  │   (IndexedDB)    │  │ (LocalStorage) │  │
│  ├────────────────────────┤  ├──────────────────┤  ├────────────────┤  │
│  │ * کدهای باندل شده Vite  │  │ * تیکت‌ها و فرم‌ها │  │ * توکن احراز   │  │
│  │ * تصاویر و آیکون‌های SVG │  │ * وضعیت مکالمات  │  │ * وضعیت آفلاین │  │
│  │ * سرفصل‌های متنی دروس   │  │ * صف خروجی سینک  │  │ * نتایج کوییز  │  │
│  └────────────────────────┘  └──────────────────┘  └────────────────┘  │
└────────────────────────────────────────────────────────────────────────┘
```

### الف) لایه تنظیمی و پیکربندی سریع (LocalStorage Wrapper)
* **موارد استفاده:** ذخیره نشست هویتی فعال دانشجویان، سوییچ شبکه شبیه‌سازی‌شده و کدهای رهگیری تخفیف.
* **دلایل استفاده:** دسترسی همزمان (Synchronous API) و سادگی کارکرد در چرخه رندرینگ هوک‌های React.

### ب) لایه داده‌های سنگین و تراکنشی (IndexedDB)
* **موارد استفاده:** ثبت سوابق ویدیوهای دیده شده توسط دانشجو، لاگ‌های تیکت‌های پشتیبانی و صف درخواست‌های هماهنگ‌نشده مشاوره.
* **مکانیسم امنیتی:** فشرده‌سازی اطلاعات با فرمت JSON و جلوگیری از دسترسی اسکریپتهای مخرب شخص‌ثالث با تکنیک Same-Origin.

## ۳. هوش مصنوعی کشینگ با Service Worker (Static & Media Caching)

معماری توآمریکا متکی بر یک Service Worker بومی با استراتژی **Cache-First** برای دارایی‌های استاتیک و **Network-First** برای روت‌های تعاملی پورتال کلاسی است.

### استراتژی کش دارایی‌های ایستای سایت (Cache-First)
```javascript
// service-worker.js (نمونه استراتژی کش منابع پایدار)
const CACHE_NAME = 'toamerica-cache-v1';
const ASSETS_TO_CACHE = [
  '/',
  '/index.html',
  '/src/main.tsx',
  '/src/App.tsx',
  '/src/index.css',
  '/metadata.json'
];

self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then((cache) => {
      return cache.addAll(ASSETS_TO_CACHE);
    })
  );
});

self.addEventListener('fetch', (event) => {
  // ترافیک روت‌های API را فیلتر کرده و استراتژی Network-First می‌زنیم
  if (event.request.url.includes('/api/')) {
    event.respondWith(
      fetch(event.request)
        .catch(() => {
          return caches.match(event.request);
        })
    );
    return;
  }

  // برای دارایی‌های استاتیک ابتدا کش محلی را چک می‌کنیم
  event.respondWith(
    caches.match(event.request).then((response) => {
      return response || fetch(event.request);
    })
  );
});
```

## ۴. چرخه همگام‌سازی امن پس‌زمینه (Secure Background Sync Lifecycle)

هنگامی که کاربر در حالت آفلاین فرم‌های مشاوره را تکمیل می‌کند یا تیکت ثبت می‌کند، سیستم مراحل زیر را برای ترابری امن طی می‌نماید:

```
 [تکمیل فرم در سفارت/آفلاین] ──> [ذخیره موقت لوکال با وضعیت Sync: false]
                                                    │
                                                    ▼
 [بررسی مجدد اتصال توسط سیستم] <── [شنود دائم تغییر آنلاین: window.addEventListener]
         │
         ├──> [در صورت آفلاین بودن] ──> [باقی ماندن در صف و نمایش هشدار زرد به کاربر]
         │
         └──> [در صورت متصل شدن شبکه] ──> [ارسال بسته‌ها به ترتیب زمانی FIFO به سرور]
                                                    │
                                                    ▼
                                    [ست شدن Sync: true و اتمام صف]
```

### گام‌های همگام‌سازی:
1. **تشخیص قطع اتصال:** استفاده از `navigator.onLine` و فرستادن هشدار به کاربر.
2. **صف‌بندی تراکنش‌ها (FIFO Queue):** ثبت متدهای ارسالی با یک تگ زمانی یکتا.
3. **همگام‌سازی تدریجی (Interval Sync):** تلاش دائم برای ارسال صف‌ها به محض شناسایی سیگنال پایدار اینترنت جهت ممانعت از سربار سروری.

## ۵. نقشه راه امنیتی و پایداری داده‌ها در تمامی شرایط (Security Roadmap)

| فاز | موضوع | مکانیسم پیاده‌سازی | هدف امنیتی |
| :--- | :--- | :--- | :--- |
| **فاز ۱** | رمزگذاری لوکال | بکارگیری کلیدهای فرعی AES-GCM درون حافظه موقت | ممانعت از سرقت فیزیکی داده‌ها از هارد دستگاه کاربر |
| **فاز ۲** | پایش پیوسته ترافیک | فعال‌سازی ردیاب‌های حجم ارسالی در هوک‌های React | ممانعت از تخریب داده به دلیل قطعی‌های ناگهانی نیم‌بند |
| **فاز ۳** | توکن یکبار مصرف | صدور OTPهای زمان‌دار محلی در حالت آفلاین | تثبیت هویت بدون ارسال دائم کلمه عبور در بستر وب |
| **فاز ۴** | مدیریت سهمیه حافظه | بررسی مداوم هشدارهای سهمیه مرورگر با `navigator.storage` | حذف هوشمند داده‌های موقت و قدیمی غیرضروری جهت عدم سرریز |

---

این نقشه راه تضمین‌کننده آن است که دانشجویان آکادمی **توآمریکا** حتی در مواقع اختلالات همه‌جانبه شبکه، تجربه کاربری روان، امن و سریعی دقیقاً مشابه اکوسیستم‌های نرم‌افزاری برتر دنیا (Apple) را در اختیار خواهند داشت.
